# Debate: What are the biggest problems and concerns in the Axiom2d codebase?

## Context

Axiom2d is a ~41,000 LOC Rust 2D game engine (12 engine crates + facade + demo) with a card game being built on top (~13,000 LOC across card_game + card_game_bin). The engine uses bevy_ecs standalone, wgpu for rendering, rapier2d for physics, and fundsp for audio. It's developed on Windows (RustRover) with Claude Code running in WSL2.

**Key structural facts:**
- Flat workspace: 14 crates under `crates/`, virtual manifest at root
- Engine scaffolding is "complete" (1100+ tests) but the card game is actively being built
- Architecture: Bevy-inspired ECS with trait-abstracted hardware (Renderer trait, AudioBackend trait, PhysicsBackend trait)
- "Code-defined assets" — no binary asset pipeline, everything is Rust code or RON
- The project is explicitly "LLM-optimized" — designed to be generated/maintained by AI
- ~114 test modules, property-based testing with proptest, snapshot testing with insta
- Heavy deps: wgpu (24), rapier2d (0.22), bevy_ecs (0.16)
- card_game has 37 pub modules with 65 pub types (high for a game library)
- Largest files: card/release.rs (2,060 LOC), splash/render.rs (1,377 LOC), card/pick.rs (1,312 LOC)
- No CI running tests against actual hardware (GPU tests are `#[ignore]`)
- All development is AI-assisted — the CLAUDE.md is 200+ lines of detailed instructions

**The question:** What are the most significant technical problems, architectural risks, and practical concerns? What should be prioritized for attention?

## Round 1

### Casey Muratori
**Position:** This codebase has traded understanding what the program actually does for an elaborate scaffolding of abstractions and tests that verify the scaffolding rather than the game.

**Arguments:**

- **The abstraction layers are not earning their keep.** There are 41,000 lines of code here and the actual game is: drag cards around a table, put them in a hand, put them in a grid. The `PhysicsBackend` trait (`physics_backend.rs`) has 14 methods, each of which needs a `NullPhysicsBackend` no-op implementation, a `SpyPhysicsBackend` in `engine_physics/src/lib.rs`, and *another* `SpyPhysicsBackend` with a different design in `card_game/src/test_helpers.rs`. That is three implementations of a 14-method interface just to test a card drag. The actual `card_drag_system` is 54 lines of physics math — the test infrastructure to verify it is larger than the code it tests. You could have just called rapier directly and had one thing to understand instead of three.

- **1,255 tests for a card game that is not yet playable.** The release profile is `opt-level = "z"` (optimize for size), which tells me this is not a performance-sensitive project, yet the test count rivals production game engines. Look at `physics_backend.rs` — there are tests like `when_set_damping_on_unknown_entity_then_no_panic` and `when_add_force_at_point_on_unknown_entity_then_no_panic`. These test that calling a no-op on a null backend does not panic. That is testing that doing nothing does nothing. Meanwhile, the GPU rendering pipeline tests are all `#[ignore]` — the thing that actually matters visually has zero automated verification in CI.

- **The Renderer trait is a textbook example of premature generalization.** `renderer.rs` defines a 16-method trait (`clear`, `draw_rect`, `draw_sprite`, `draw_shape`, `draw_text`, `set_view_projection`, `set_blend_mode`, `set_shader`, `set_material_uniforms`, `bind_material_texture`, `compile_shader`, `upload_atlas`, `viewport_size`, `apply_post_process`, `present`, `resize`). This is essentially a re-declaration of the entire GPU API surface. There is exactly one real implementation (`WgpuRenderer`) and two fake ones (`NullRenderer`, `SpyRenderer`). The `SpyRenderer` alone has 11 optional capture fields (`color_capture`, `rect_calls`, `sprite_calls`, `shape_calls`, `matrix_capture`, `blend_calls`, `shader_calls`, `uniform_calls`, `texture_bind_calls`, `compile_shader_calls`, `text_calls`), each wrapped in `Arc<Mutex<Vec<_>>>`. That is not a useful abstraction — that is a testing bureaucracy. When you need to add a new rendering feature, you now have to update three structs and 16 method signatures. The trait boundary should have been at a much higher level (like "render a frame") or not existed at all.

- **The crate structure is organizational, not architectural.** 14 crates, but there is no compilation boundary that matters. `card_game` depends on `engine_physics`, `engine_render`, `engine_input`, `engine_core`, `engine_scene`, `engine_app`, `engine_ecs`, and `engine_ui`. Changing `engine_core` recompiles everything. The flat crate layout gives the *appearance* of modularity but the dependency graph is a funnel. A struct change in `engine_core/types.rs` (where `Pixels`, `Seconds`, `TextureId` live) cascades through every crate. This is not modularity; it is ceremony.

- **NewType wrappers add friction without adding safety in practice.** `Pixels(f32)` and `Seconds(f32)` are used as newtype wrappers, but look at the actual game code in `card_game/src/card/geometry.rs` — `TABLE_CARD_WIDTH: f32 = 60.0`, `ART_HALF_W: f32 = 27.0`. The card game code just uses raw `f32` everywhere. The newtypes are only enforced at the Renderer boundary (`Rect` uses `Pixels`), so you have a type-safety wall in the one place where it does not prevent bugs (the renderer takes whatever you give it) and raw floats in the places where dimensional confusion would actually hurt (physics coordinates, screen coordinates, world coordinates mixing in `stash/render.rs`).

**Challenges to others:**

- **To Kent Beck (anticipated TDD defense):** I expect Kent to say the 1,255 tests give confidence to refactor. But confidence in *what*? The tests are tightly coupled to the implementation — `SpyRenderer` captures verify the exact sequence of draw calls, `SpyPhysicsBackend` captures verify exact method calls. These are not behavior tests; they are implementation mirrors. When you refactor the rendering to batch differently, you break the spy tests. When you change the physics approach, you break the spy tests. The tests are not enabling refactoring — they are preventing it by encoding the current implementation as a specification.

- **To Rich Hickey (anticipated simplicity argument):** Rich will likely argue the crate structure creates unnecessary complexity. I agree, but I would go further: the *entire ECS architecture* is unnecessary complexity for this game. This is a card game with at most a few dozen entities. You do not need archetypal storage, you do not need system scheduling with five phases, you do not need `SystemParam` bundles. A struct with `cards: Vec<CardState>`, a `hand: Vec<CardId>`, and a `stash: [[Option<CardId>; 10]; 10]` plus a render function would be a fraction of the code and vastly easier to understand. The ECS is here because the engine was designed before the game, not because the game demanded it.

**What I might be wrong about:**

- The "LLM-optimized" angle changes the calculus in a way I do not have experience with. If the primary developer is an AI that benefits from highly structured, predictable patterns (trait + null impl + spy impl + system function + tests), then the boilerplate I am criticizing might be the *correct trade* — the cost of the boilerplate is borne by a machine that does not get bored or confused, and the benefit is that the AI can reliably extend the codebase by pattern-matching against existing code. The rigid structure might be a feature, not a bug, when your developer literally works by pattern completion. I still think the result is worse than what a human who understands the hardware would write, but I acknowledge the development model is different enough that my heuristics about "unnecessary abstraction" may not apply in the same way.

### Bryan Cantrill
**Position:** This codebase has a catastrophic observability vacuum — there is zero runtime instrumentation, every hardware trait silently swallows failures, and when something goes wrong in production you will have absolutely no way to understand what happened.

**Arguments:**

- **Zero logging, zero tracing, zero metrics — in a 41,000 LOC systems codebase.** I searched every crate. The entirety of runtime diagnostic output in this engine is two `eprintln!` calls: one in `crates/engine_audio/src/backend/cpal.rs:61` for audio stream errors, and one in `crates/engine_render/src/wgpu_renderer/renderer_trait.rs:406` for surface errors. That's it. There is no `log` crate, no `tracing` crate, no structured logging, no counters, no histograms, nothing. You cannot ask this system "what are you doing?" at runtime. You cannot distinguish between "the system is working correctly but slowly" and "the system is silently dropping every draw call." This is not a theoretical concern — this is the difference between being able to diagnose a problem and staring at a black screen wondering what's happening. In DTrace terms, this system has zero probes. You're flying blind.

- **Every hardware trait returns nothing and reports nothing on failure.** Look at the `Renderer` trait in `crates/engine_render/src/renderer.rs`. Every single method returns `()`. `draw_rect`, `draw_sprite`, `draw_shape`, `upload_atlas`, `compile_shader`, `set_view_projection` — all void. If the GPU rejects your shader, you get nothing. If atlas upload fails, you get nothing. If draw calls are silently dropped because the renderer is in a bad state, you get nothing. The `PhysicsBackend` trait has the same problem — `set_linear_velocity`, `set_angular_velocity`, `add_force_at_point`, `set_damping`, `set_collision_group` all return `()`. The `AudioBackend::stop()` returns `()`. You call `stop(PlaybackId(999))` on a sound that doesn't exist and nothing happens. These aren't just "we'll add error handling later" — the trait definitions themselves have baked in the assumption that operations cannot fail, which means every implementor is forced into silent failure by the API contract.

- **The `NullRenderer`, `NullPhysicsBackend`, and `NullAudioBackend` are indistinguishable from broken implementations.** A `NullRenderer` returns `(0, 0)` for viewport size. A `NullPhysicsBackend` returns `None` for every position query. What does the game code do when it gets these values? It silently proceeds. In `physics_sync_system.rs`, when `body_position()` returns `None`, the transform is simply not updated — no warning, no counter, no indication that physics is not running. In `card_release_system`, `viewport_size()` returning `(0, 0)` means `vh` is `0.0`, which means `is_hand_drop_zone` will never return true (line 23-25 of release.rs: `screen_y >= viewport_height - HAND_DROP_ZONE_HEIGHT` with viewport_height 0.0). The game just silently behaves wrong. This is the kind of failure mode that makes you lose days.

- **The release profile is configured to destroy all diagnostic capability.** In the root `Cargo.toml`: `strip = true` (removes all symbols and debug info), `panic = "abort"` (no unwinding, no backtrace), `opt-level = "z"` (size optimization over debuggability). If you ship this binary and something goes wrong, you get a process that aborts with no symbols, no stack trace, no structured logs. You literally cannot post-mortem this. The combination of "no runtime instrumentation" and "no crash-time instrumentation" means your only debugging tool is "try to reproduce it in debug mode and add printlns."

- **The `EngineError` type exists but is almost never used.** `crates/engine_core/src/error.rs` defines `EngineError` with `NotFound` and `InvalidInput` variants. I searched for actual usage of this type across the engine — it barely appears in function signatures. The Renderer trait methods don't return `Result`. The PhysicsBackend methods don't return `Result`. The engine has a canonical error type that nobody uses because the trait APIs were designed to return `()`. The error type is a monument to good intentions, abandoned by the architecture.

- **The card game's `.expect()` messages are the only documentation of invariants, and they'll become panics in production.** In `card_release.rs:190`, `grid.place(...).expect("slot should be empty: guarded by is_none check above")`. In `plugin.rs`, there are six `.unwrap()` calls during plugin initialization. These encode important invariants as panic messages rather than as proper error handling paths. With `panic = "abort"` in release, these become instant process termination with no recovery path and (with `strip = true`) no useful diagnostic information.

- **The `CpalBackend` silently degrades to no audio with no indication.** Look at `crates/engine_audio/src/backend/cpal.rs:44-69`. The `open_stream` method returns `Option<cpal::Stream>`. If the host has no output device? Returns `None`. If `default_output_config()` fails? Returns `None`. If sample format isn't `F32`? Returns `None`. If `build_output_stream` fails? Returns `None`. If `stream.play()` fails? Returns `None`. The `CpalBackend` stores `_stream: Option<StreamHandle>`, and if it's `None`, the backend is constructed successfully but silently produces no audio whatsoever. Every `play_on_track` call will still succeed, return a `PlaybackId`, push to `active_sounds`, and the sounds will never be heard. The user sees: "I called play and got an ID back but there's no sound." There is nothing in this architecture that distinguishes "audio is working" from "audio device was never opened."

**Challenges to others:**

- **To Casey Muratori:** Casey argues the ECS is unnecessary for a card game and you could replace it with a simple struct. He's right about the overhead, but wrong about the prescription. The ECS isn't the problem — it's a reasonable architectural choice for a game engine that might support more than one game. The problem is that the abstractions *within* this ECS architecture were designed to be opaque fire-and-forget calls. Casey wants fewer abstractions. I want the existing abstractions to be *transparent* — to tell you what's happening inside them. You can have a trait-based renderer and still have it report "I dropped 3 draw calls because the surface was lost" or "atlas upload took 14ms." The abstraction layers aren't the disease; the lack of observability through those layers is.

- **To Kent Beck:** I expect Kent to celebrate the testing discipline here — the Arrange/Act/Assert pattern, the SpyRenderer, the property-based testing. And I agree the test infrastructure is genuinely well done. But here's the problem: every SpyRenderer test verifies "this system called these methods in this order." That tells you the code does what you expect *when you're watching*. What about when you're not? The SpyRenderer is observability for test time. There is literally nothing analogous for runtime. The gap between "we can observe the system in tests" and "we can observe the system in production" is total.

**What I might be wrong about:**

- It's possible that for a game (as opposed to a server or an OS kernel), the observability gap matters less than I think. Games have a tight development loop — you're playing the game constantly, visually verifying behavior, and most bugs manifest as visible glitches rather than silent data corruption. The "production" environment for a game is quite different from a distributed system. A developer staring at the screen *is* a form of observability. The lack of structured logging might be entirely acceptable for a project at this stage, and adding tracing infrastructure to a 2D card game engine might genuinely be premature. Maybe the right answer is "add `eprintln!` when you need it and move on." But I don't think so — the silent failure patterns in the trait APIs will compound, and the first time someone spends a day debugging why cards won't drop into the hand zone because the viewport size was `(0, 0)`, they'll wish they had a single log line.

### Kent Beck
**Position:** The biggest problem in Axiom2d is not missing features or performance debt -- it is that the tests have stopped being a design tool and have become a bureaucratic obligation, and this is hiding the real risk: nobody knows if this card game is actually fun.

**Arguments:**

- **The tests are drowning in ceremony, not driving design.** In `card/release.rs`, there are 41 tests across 2,060 lines. The average test has 29 lines of Arrange, 3 lines of Act, and 10 lines of Assert. The `DragState { dragging: Some(DragInfo { entity, local_grab_offset: Vec2::ZERO, origin_zone: CardZone::Table, stash_cursor_follow: false, origin_position: Vec2::ZERO }) }` block is copy-pasted 40 times. `Collider::Aabb(Vec2::new(30.0, 45.0))` appears 39 times in that file alone. This is not testing driving design -- this is testing following behind a design that has already been decided, and doing so mechanically. When your Arrange section is 10x your Act section, the test is telling you the API is too complicated to use. But nobody is listening.

- **Tests that should be one test are split into many.** Look at `when_release_in_hand_area_from_table_then_card_added_to_hand`, `when_release_in_hand_area_from_table_then_zone_becomes_hand`, `when_release_in_hand_area_from_table_then_render_layer_becomes_ui`, `when_release_in_hand_area_from_table_then_physics_body_removed`. Four tests, identical Arrange, identical Act, each checking one field. One test named `when_card_released_into_hand_then_state_fully_transitions` with four assertions would be clearer, shorter, and provide the same diagnostic value. The one-assertion-per-test dogma is producing enormous duplication without added clarity. My rules of simple design say "no duplication" -- test code counts.

- **357 tests in card_game but zero end-to-end gameplay tests.** Every test runs exactly one system in isolation against a manually constructed World. The `CardGamePlugin` wires 15+ systems together with `.chain()` ordering constraints, but not a single test exercises the actual pick-drag-release cycle through the real schedule. The plugin tests in `plugin.rs` only check that resources got inserted -- they do not run a single frame. The feedback loop is broken: these tests tell you each gear turns, but not that the clock keeps time. If the system ordering in `register_systems()` is wrong, no test catches it.

- **The roadmap is all DONE but there is no game loop.** Every step in `Doc/Card_Game_Roadmap.md` is marked `[DONE]`. All 8 phases complete. But `card_game_bin/src/main.rs` is 107 lines that spawn some cards and apply two impulses. There is no win condition, no turn structure, no scoring, no opponent. The entire effort has been infrastructure. The biggest risk in any software project is building the wrong thing. The question "is dragging cards around a table with spring physics actually engaging?" has not been answered after 12,370 lines of card_game code. That is the real debt.

- **The feedback loop between player and developer does not exist.** There is no playtest data, no way to record and replay a session, no telemetry, no screenshots of gameplay. The `#[ignore]` on GPU tests means the actual visual output has never been verified in CI. The project has invested heavily in unit-level correctness but has no mechanism to answer "does this feel good?" -- which for a physics-based card game is THE question.

- **65 public types in the card_game library crate is a design smell.** The prelude re-exports 37 symbols. `CardGameState` bundles 5 resources. The card module alone has 16 submodules. For a game that currently just drags cards around, this is premature structure. Simple design says: fewest elements. You do not need 16 submodules for drag-and-drop.

**Challenges to others:**

- **To Casey Muratori, on spy tests encoding implementation:** Casey correctly identifies that SpyRenderer and SpyPhysicsBackend tests are "implementation mirrors." I agree completely -- these tests are fragile and will break on any refactor. But Casey's proposed solution of removing the ECS and writing plain structs would not fix the testing problem; it would just move it. The real issue is that the test strategy never evolved past "verify this system calls these methods." What is missing is behavior-level testing: "when I click a card and drag it to the hand zone, the card ends up in the hand." That test should work regardless of whether the implementation uses ECS systems, plain functions, or a magic 8-ball. The abstractions are not the root cause of bad tests -- the testing strategy is.

- **To Bryan Cantrill, on observability:** Bryan makes a strong case that the trait APIs swallow failures silently. I agree this is real debt. But I would argue the tests themselves are *also* a form of broken observability. When a test has 29 lines of Arrange that all look the same, it becomes invisible -- the developer skims past it. The test output says "357 passed" but communicates nothing about what those tests actually verified. A test suite should be a readable specification of what the system does. This one is unreadable due to copy-paste volume. Fix the tests first, then add runtime observability -- because the tests are the observability you use every single day during development.

**What I might be wrong about:**

- The team might be intentionally building a game engine toolkit where the "card game" is really a tech demo, not a product. If the goal is to prove the engine works, then thorough unit tests of each system in isolation is exactly the right strategy, and the absence of gameplay is acceptable. The 357 tests become a comprehensive regression suite for an engine feature showcase. I would still push for less duplication in the test code, but I would stop complaining about the missing game loop. The question is whether the team knows which of these two things they are building.

### Rich Hickey
**Position:** The biggest problem in Axiom2d is pervasive complecting of identity, state, and value — the codebase has chosen "easy" (familiar ECS patterns, mutable resources, imperative state transitions) over "simple" (separated concerns, data as values, declarative descriptions).

**Arguments:**

- **State and identity are complected everywhere via mutable resources.** Look at `CardGameState` in `card/game_state_param.rs`: it bundles `ResMut<DragState>`, `ResMut<Hand>`, `ResMut<PhysicsRes>`, `Res<StashVisible>`, `ResMut<StashGrid>` into a single SystemParam. Five unrelated concerns — drag tracking, inventory, physics, visibility, grid storage — are coupled by being co-accessed. The `card_release_system` in `card/release.rs` (2,060 LOC) needs `CardGameState`, `RendererRes`, `Commands`, and two queries because releasing a card must imperatively mutate state across six orthogonal subsystems. The zone transition logic in `drop_on_hand`, `drop_on_stash`, `drop_on_table` manually orchestrates removing physics bodies, inserting components, updating resources, and changing zones. This is place-oriented programming. A card moving from table to hand should be a *value* — a zone transition event — that other systems react to independently. Instead it is an imperative procedure reaching into physics, render layers, item forms, zone components, hand resources, commands, and collision groups simultaneously.

- **Card entity is a bag of complected concerns, not information.** A single card entity carries: `Card`, `CardDefinition`, `CardLabel`, `CardZone`, `Transform2D`, `RigidBody`, `Collider`, `RenderLayer`, `SortOrder`, plus 7-9 child entities (4 front face shapes, 2 back shapes, 2 text children, 1 stash icon). The Card's *identity* (its definition) is complected with its *representation* (shapes, labels, stash icon) and its *location* (physics body, zone, render layer). When you drag a card from the stash to the table, `drop_on_table` must imperatively add `RigidBody::Dynamic`, remove `CardItemForm`, change `RenderLayer`, call `activate_physics_body`, insert `ScaleSpring`, and update `CardZone`. Six mutations across six distinct subsystems for the single fact: "this card is now on the table." The rendering and physics consequences should be *derived* by separate systems reacting to the zone change, not commanded by an orchestration function.

- **The PhysicsBackend trait is a place, not a value.** Fourteen methods, all `&mut self`. You cannot snapshot, diff, or replay the physics world. The `SpyPhysicsBackend` in `card_game/src/test_helpers.rs` has 8 `Arc<Mutex<Vec<_>>>` fields — `AddBodyLog`, `ColliderLog`, `RemoveBodyLog`, `DampingLog`, `VelocityLog`, `AngularVelocityLog`, `CollisionGroupLog`, plus positions/rotations maps. That is the tax for testing against a place. If physics interactions were expressed as values — command structs going in, fact structs coming out — testing would be comparing data, not spying on method invocations through synchronized log vectors.

- **Data is encoded as component presence/absence rather than as information.** `CardItemForm` is a zero-sized marker whose presence means "stash icon mode." `StashIcon`, `FlipAnimation`, `HandSpring` — all markers. The actual visual state of a card is distributed across the presence of `CardItemForm`, the `Visible` flags on 9 child entities, the `CardZone::Stash` variant, and `RenderLayer::UI`. The `card_item_form_visibility_system` exists solely to synchronize these scattered representations. If the visual mode were a single enum value (`TableFaceUp | TableFaceDown | StashIcon | HandCard`), there would be nothing to synchronize — you would just match on the enum and derive the rest. This is the classic mistake of encoding information as the *arrangement* of mutable places rather than as data you can inspect.

- **The module explosion is a symptom, not the disease.** 37 pub modules, 65 pub types — `face_layout.rs` (85 LOC), `face_side.rs`, `geometry.rs`, `label.rs`, `art_shader.rs`, `item_form.rs`, `render_layer.rs`, `physics_helpers.rs`. Each is a tiny slice of card behavior with its own test module. This creates the *appearance* of separation but the *reality* of entanglement. The proof: `pick.rs` (1,312 LOC) and `release.rs` (2,060 LOC) are enormous precisely because zone transitions must reach across all these scattered modules to coordinate mutations. The files are small individually, but the system's behavior cannot be understood without reading all of them. That is not simplicity. That is complexity distributed across many files so it looks organized.

**Challenges to others:**

- **To Casey Muratori:** Casey argues the entire ECS is unnecessary and a flat `Vec<CardState>` plus a render function would suffice. He is right about the entity count being trivial, but his prescription would recreate the same complecting problem. Whether you express "card moves from stash to table" as imperative mutations on ECS components or as imperative mutations on struct fields, you still have the same coordination problem — you still need to update physics state, visual state, inventory state, and animation state in lockstep. Casey's solution removes the framework tax but preserves the fundamental architectural defect. The issue is not ECS overhead; it is that zone transitions are not modeled as data that flows through the system.

- **To Bryan Cantrill:** Bryan correctly identifies the silent-failure, void-returning trait pattern. But I would reframe his diagnosis: the observability vacuum is a *symptom* of place-oriented design, not the root cause. When your entire architecture is "mutate this place, then that place, then the other," of course there is nothing to observe — there are no values flowing through the system. You cannot log what does not exist as data. If zone transitions were values (events), you could trivially log them. If physics commands were a command buffer of structs, you could inspect them. If render operations were a draw list, you could dump it. Bryan's prescription — add logging and tracing — treats the symptom. Making operations into data would make observability nearly free, because you would have something concrete to observe.

- **To Kent Beck:** Kent correctly notes the test duplication and argues for extract-and-parameterize. But the test problem runs deeper than style. In `card/definition.rs`, 19 of 21 tests are RON serialization roundtrips — testing that serde's derive macro works. In `spawn_table_card.rs`, 591 lines of tests verify structural properties like "front face has four shape children" and "back center pattern uses thirty percent." These encode the current implementation as specification. When the card layout changes, every test breaks not because behavior is wrong but because the specification was the implementation. Kent's refactoring advice is good but insufficient: the test *strategy* needs to move from "verify implementation structure" to "verify behavioral outcomes." No amount of extracting helper methods fixes a test that asserts the wrong thing.

**What I might be wrong about:**

- Rust's ownership model and bevy_ecs's design may genuinely constrain how far you can push data-orientation. In Clojure, I would have zone transitions be pure functions from old state to new state, with persistent data structures making this cheap. In Rust with bevy_ecs, the system signature determines resource access, and the borrow checker forbids holding multiple mutable references. The `CardGameState` SystemParam bundle may be the pragmatic minimum within bevy_ecs's constraints — separate systems communicating through event buffers would add more types, more ordering constraints, and more subtle bugs. It is possible that what I call "complecting" is actually the least bad coordination pattern available, and that the 2,060-line release system, while ugly, is mechanically correct and locally comprehensible in a way that a distributed event-driven approach would struggle to match.

## Round 2

### Casey Muratori
**Position:** The architecture's central failure is that it optimizes for the appearance of engineering rigor — test counts, crate counts, abstraction layers — while avoiding the hard question of whether the program works, and the other panelists' prescriptions (observability, data-orientation, test strategy) are treating symptoms of a project that should not have this much code in the first place.

**Responses to challenges:**

- **To Bryan Cantrill ("abstractions aren't the disease; lack of observability is"):** Bryan, I actually agree with you more than I expected to after reading the code again. You are right that you can have a trait-based renderer that reports failures — I was conflating two problems. The trait boundary itself is not the sin; the void-returning API contract is. Look at `drop_on_hand` in `release.rs` line 156: `physics.remove_body(entity)` returns nothing. If that entity was never in the physics world, nothing happens. Then line 157: `hand.add(entity)` returns `Result`, and the code *does* handle the `Err` case by snapping back. So we know the author understands fallible operations — they just chose to make the physics API infallible. That inconsistency is more damning than the trait's existence. You could keep the trait, make `remove_body` return `Result`, and the problem goes away. I still think the trait has too many methods (14 on PhysicsBackend, 16 on Renderer), but you changed my mind that the shape of the API matters more than whether there is a trait at all.

- **To Rich Hickey ("Vec<CardState> would recreate the same complecting problem"):** You are half right. Yes, if I just flattened the ECS into a struct and kept the same imperative "reach into six subsystems" pattern, I would have the same coordination problem. But here is where I disagree: the coordination problem exists *because* the architecture chose to distribute card state across independent systems in the first place. ## Synthesis

### Genuine Consensus

1. **The system ordering in `register_systems()` is the highest-consequence untested thing.** Casey identified it in Round 2 ("the plugin registration is the real specification, and it has zero tests"), Bryan sharpened it ("the ordering is not just untested — it is unobservable"), and Kent called it "the single most important concrete finding in this debate." Rich implicitly agreed by noting that the imperative zone transitions depend on ordering that is "correct by convention" not by construction. All four experts converged on this as the most dangerous gap.

2. **The test suite includes a significant fraction of tests that verify framework guarantees rather than authored behavior.** Rich identified serde roundtrips and derive-verification tests. Kent counted them and noted they violate the project's own CLAUDE.md rules. Casey called the spy-based tests "implementation mirrors." All four agree that test count is creating false confidence — the disagreement is only on what percentage is waste (Kent estimates 30-40% of card_game tests could be deleted with zero coverage loss).

3. **The void-returning hardware trait APIs (`Renderer`, `PhysicsBackend`, `AudioBackend`) are a design defect.** Casey initially argued for removing the traits entirely but was moved by Bryan to concede "the void-returning API contract is more damning than the trait's existence." Bryan provided concrete failure modes (`CpalBackend` silently degrading, `viewport_size()` returning `(0, 0)` making hand drops permanently unreachable). Rich agreed the hardware boundary is a contract problem independent of his data-orientation argument. Kent did not dispute this point. All four accept these traits need `Result` return types.

4. **There are zero end-to-end gameplay tests.** Kent named it first ("357 tests but zero end-to-end gameplay tests"), Casey reinforced it ("nobody tested that the clock tells time"), Bryan extended it to runtime observability. Rich agreed indirectly by noting the tests "encode implementation as specification." All four converge: no test exercises the pick-drag-release cycle through the real schedule.

5. **The release profile (`strip = true`, `panic = "abort"`, `opt-level = "z"`) destroys diagnostic capability.** Bryan raised this in both rounds. No other expert disputed it. Kent's Round 2 acknowledged that Bryan's `drop_on_hand` walkthrough — where a despawned entity causes untraceable process death — was compelling. This is consensus by non-objection from Casey and Rich, and explicit agreement from Kent.

### Unresolved Tensions

**1. ECS: keep, reform, or remove?**

- **Casey (remove):** Fifteen cards do not need archetypal storage, five-phase scheduling, or SystemParam bundles. A flat struct with `cards: Vec<CardState>` and a render function would be a fraction of the code. The ECS *enables* the proliferation of bad tests by providing too many seams. Rich's event-driven alternative would double the system count without eliminating coordination.
- **Rich (reform via data-orientation):** The ECS itself is the wrong tool (conceded in Round 2), but Casey's flat-struct alternative would recreate the same complecting problem — imperative mutations across physics, render, and inventory state. The fix is a `ZoneConfig` table driven by the existing `CardZone` enum, with a reconciliation system that computes diffs. This is easier without the ECS but is not *about* removing the ECS.
- **Bryan (keep but make transparent):** The ECS is a reasonable architectural choice for an engine. The problem is that the abstractions within it are opaque. Keep the traits, add `Result` returns, add logging. The discipline problem (bad tests, void APIs) is cultural, not architectural.
- **Kent (architecture is secondary to test strategy):** Whether you use ECS, flat structs, or event sourcing, the test strategy must change independently. TDD pulls design toward testability; the test-after workflow produced the current over-seamed design. The seam count is a consequence, not a cause.

**Strongest tension:** Casey and Rich agree the ECS is wrong for this game but disagree on the replacement (flat imperative vs. data-driven reconciliation). Bryan and Kent say the architecture is secondary to the API contracts and test strategy respectively. This is a genuine fork: do you rewrite the game layer's architecture, or do you fix the contracts and tests within the current architecture?

**2. Imperative zone transitions: correct-by-construction or accident-waiting-to-happen?**

- **Casey (imperative is pragmatically correct):** The seven mutations in `drop_on_hand` have causal ordering. The imperative version is "correct by construction because the ordering is lexical." Rich's event-driven decomposition would encode the same ordering implicitly in schedule constraints, which is strictly harder to verify.
- **Rich (imperative ordering is illusory):** Lexical ordering gives no guarantees — it gives accidents that happen to work. Nothing prevents a future developer from reordering lines. A reconciliation system reading desired state and computing deltas has no ordering to get wrong. Casey identified a causal ordering that is an artifact of the imperative style, not a property of the domain.

**3. How much of the test suite should be deleted vs. refactored?**

- **Kent (delete 30-40%, then write behavior tests):** Many tests are structurally incapable of surviving a refactor. The right response to a test asserting the wrong thing is to delete it and ask what question you need answered. "Fewest elements" means eliminating tests that verify nothing useful.
- **Rich (deletion is necessary but insufficient):** The tests are wrong because the codebase has no concept of behavioral outcome to assert against. There is no `is_card_in_hand(entity) -> bool` predicate. You need new abstractions (like `ZoneConfig`) that simultaneously serve as test specifications and runtime data. Deletion without replacement leaves you with fewer tests and the same blind spots.

### Blind Spots

1. **No expert addressed the LLM development workflow beyond criticizing its outputs.** All four noted that the LLM generates pattern-matching tests and produces boilerplate. Nobody proposed how to *fix the LLM's instructions* to produce better results. The CLAUDE.md is 200+ lines of detailed process — it could be updated to ban serde roundtrip tests, require at least one end-to-end test per step, or mandate behavior-level assertions. The debate treated the LLM as a force of nature rather than a configurable tool.

2. **Nobody discussed the card game's actual design or gameplay.** Kent came closest ("nobody knows if this card game is actually fun") but no expert engaged with what the game *should be*. There is no win condition, no turn structure, no scoring. The entire debate focused on code quality of infrastructure for a game that does not yet exist as a game. The question of whether the physics-based card manipulation is the right core mechanic was never examined.

3. **No expert addressed dependency risk.** The project depends on `bevy_ecs 0.16`, `wgpu 24`, `rapier2d 0.22` — all fast-moving crates with breaking changes between versions. The tight coupling to `bevy_ecs` internals (SystemParam derives, Schedule types) means a bevy_ecs version bump could require touching every crate. Nobody discussed version pinning strategy, upgrade cost, or whether the engine's thin wrappers actually insulate against upstream churn.

4. **Performance was never seriously discussed.** Casey mentioned `opt-level = "z"` as evidence this is not performance-sensitive, but nobody examined whether the physics-based card dragging (rapier2d for 15 cards) has acceptable frame timing, whether the five-phase schedule has measurable overhead, or whether the tessellation-per-frame approach in shape rendering is a problem. For a game with spring physics and drag interactions, frame latency directly affects feel.

5. **The Windows/WSL2 development environment was not examined.** The project runs `cargo.exe` (Windows Rust) from WSL2, with the repo on a `/mnt/c` drvfs mount. Build times, file I/O performance, and the inability to run GPU tests in CI are all shaped by this setup. No expert asked whether this environment itself is a constraint that should be addressed.

### Prioritized Recommendations

1. **Write 3-5 end-to-end tests that exercise the real schedule.** (Consensus: Casey, Bryan, Kent, Rich all identified this gap.) Create tests that construct a `World`, register all systems via `CardGamePlugin`, and run multiple frames simulating pick-drag-release through the actual schedule. This is the single highest-value testing investment because it verifies system ordering — the one thing all experts agree is the most dangerous untested property. Accept that these tests will be slower than unit tests.

2. **Add `Result` return types to hardware trait methods.** (Consensus: Casey, Bryan, Rich.) Start with `PhysicsBackend::remove_body` and `Renderer::upload_atlas` / `compile_shader` — the methods where silent failure has concrete downstream consequences (Bryan's `viewport_size() -> (0,0)` and `CpalBackend` examples). This is a breaking API change but the codebase has exactly one real implementation of each trait, so the blast radius is contained. Propagate errors or log-and-continue at call sites — do not `.unwrap()`.

3. **Delete tests that verify framework guarantees.** (Consensus: Kent, Rich; Casey agrees implicitly.) Audit `card_game` tests for: serde roundtrip tests on derived types, PartialEq/Debug derive verification, constructor-stores-arguments tests, and Default-returns-default tests. Kent estimates 30-40% of 357 tests. This is directly actionable: the CLAUDE.md already bans these test categories but the ban is not enforced. Update CLAUDE.md to explicitly ban serde roundtrip tests on types that only use derived impls.

4. **Extract test setup into shared builders and collapse single-assertion test families.** (Kent's primary argument, Bryan concurred.) The 41 tests in `release.rs` with 29-line Arrange blocks copied 40 times should use a builder pattern (`ReleaseTestBuilder::card_on_table().with_hand().build()`). The four `when_release_in_hand_area_from_table_then_*` tests should be one test with four assertions. This reduces noise and makes the remaining tests readable as specification.

5. **Soften the release profile for diagnostics.** (Bryan, unopposed.) Change `strip = true` to `strip = "debuginfo"` (keeps symbol names, strips DWARF), or at minimum remove `strip = true` until the project has runtime logging. Consider `panic = "unwind"` in a `[profile.release-debug]` profile for diagnostic builds. This is zero-cost during development and recoverable in production.

6. **Add minimal structured logging at trait boundaries.** (Bryan's primary argument, Kent concurred on sequencing.) Add the `tracing` crate. Instrument hardware trait implementations (not the trait itself) with `tracing::warn!` on failure paths and `tracing::debug!` on significant operations (atlas upload, physics step, audio stream open). This gives Bryan's observability without Rich's architectural rewrite. Start with `CpalBackend` (silent degradation is the most concrete diagnosed failure) and `WgpuRenderer` surface errors.

7. **Introduce a `CardZone`-driven reconciliation pattern for new zone transitions.** (Rich's proposal, Kent endorsed the direction.) Do not rewrite existing `drop_on_*` functions — they work. For the *next* zone or transition added, implement Rich's `ZoneConfig` table pattern and a reconciliation system. This lets you evaluate the approach incrementally without a rewrite. If it proves cleaner, migrate existing transitions one at a time.

8. **Update CLAUDE.md to prevent test disease propagation.** (Rich identified the mechanism, Kent diagnosed the LLM-specific risk.) Add explicit rules: no serde roundtrip tests on types with only derived Serialize/Deserialize; every roadmap step must include at least one multi-system or end-to-end test; spy-based tests must assert behavioral outcomes (card is in hand) not method call sequences (remove_body was called). This addresses the root cause that all four experts identified: the LLM generates tests by pattern-matching against existing examples.

### What Trade-offs Are Being Accepted

- **Keeping the ECS for now.** Casey and Rich both argue it is the wrong tool for this game. The recommendations above do not include removing bevy_ecs because Bryan and Kent argue (and Casey partially conceded) that the contracts and test strategy matter more than the framework choice, and an ECS rewrite would consume weeks while delivering no gameplay. The accepted cost: continued SystemParam ceremony, five-phase scheduling overhead for 15 entities, and the framework tax that Rich identified. This trade-off should be revisited if the card game grows beyond a single-screen experience.

- **Not pursuing Rich's full data-oriented redesign.** The `ZoneConfig` reconciliation pattern is recommended only for new code (item 7). The existing imperative `drop_on_*` functions remain. The accepted cost: continued coordination complexity in `release.rs`, continued risk of ordering bugs when modifying zone transitions, and spy-based tests on those paths until they are migrated. This is accepted because the imperative code works today and a full rewrite without end-to-end tests (item 1) to catch regressions would be reckless.

- **Not addressing gameplay or game design.** Kent's most important strategic point — "nobody knows if this card game is actually fun" — is not addressed by any recommendation. The accepted cost: continued investment in infrastructure for a game that may not be engaging. This is accepted because the recommendations above are prerequisites for being *able* to iterate on gameplay (you need end-to-end tests and observable runtime behavior before you can playtest and adjust), but the team should be aware that the gameplay question remains unanswered after 12,370 lines of code.

- **Not reducing crate count or public API surface.** Casey's argument about the 14-crate funnel dependency graph and Rich's argument about 65 public types / 37 prelude symbols are acknowledged but not acted on. The accepted cost: continued compile cascade from `engine_core` changes, continued cognitive overhead from the large public surface. This is deferred because the crate structure is stable and working — restructuring it is high-effort, low-immediate-payoff compared to the testing and observability gaps.
